# Вопросы и ответы

## 1. Что может относится к gc roots?

1. Локальные переменные / ссылки в стеке активного потока
2. Статические ссылки(поля) доступные из загруженных классов
3. JNI ссылки, например GlobalRef
4. Любая запись в HashMap
5. Активные потоки Thread в приложении
6. Объекты, на которые ссылаются из нативного когда через JNI WeakGlobalRef
7. Объекты на которые ссылаются из WeakReference
8. Любой объекты, находящийся в поколении Old Gen
9. Аргументы метода main
10. Любой объект, созданный через new, пока не вызван GC

## 2. Что может вызывать утечку памяти?

1. Объявленный inner class с захватом внешнего объекта
2. Lambda, которая захыватывает this и сохраняется на уровне свойств класса
3. Handler, которому назначили задачу с методом postDelayed()
4. Ссылка на activity в Presenter-е, обернутая в WeakReference

## 3. Что будет, если разработчик вызовет метод System.gc() при закрытии приложения?

1. GC выполнится немедленно и гарантированно
2. JNI может проигнорировать, а частые вызовы могут ухудшить производительность
3. System.gc чистит stack - это небезопасно
4. System.gc удаляет все weak references моментально

## 4. Что-то про SoftReference

1. SoftReference очищается при первой же сборке мусора
2. Очистка SoftReference происходит, если в приложении закончилась память
3. Поведение SoftReference зависит от политики сборщика мусора и деления памяти, поэтому кэш может очищаться непредсказуемо
4. SoftReference никогда не очищается
5. SoftReference запрещен в Android
6. SoftReference делает кэш детерминированным: объект будет храниться, пок на него есть ссылка SoftReference

## 9. Какие утверждения являются верными

1. Deadlock возможен только при использовании (???), а не Mutex
2. Race condition может возникнуть даже если программа не падает и иногда работает правильно
3. Атомарность гарантирует отсутствие проблем видимости памяти
4. Visibility problem возникает, когда изменение сделанные одним стеком не всегда видны другому
5. Volatile гарантирует атомарность сложных операций, например, counter++


## 10. Вы находитесь на активити A, есть кнопка перехода к активити B, после перехода на активити B нажимается системная кнопка Back, какие утверждения верны?

1. У А гарантированно вызовется onPause() -> onStop() при открытии B
2. У А всегда будет вызыван onDestroy() при открытии B
3. B при старте проходит onCreate() -> onStart() -> onResume()
4. B проходит onPause() -> onStop() -> onDestroy() при нажатии Back в B
5. А проходит onRestart() -> onStart() -> onResume() при нажатии Back в B
6. A может получить только onResume() без onStart() при нажатии Back в B, если A не была остановлена
7. Если B прозрачная/диалоговая, то A при открытии B все равно будет остановлена onStop() обязателен
8. В режиме multi-window/картинка в картинке A при открытии B может остаться в RESUMED и не получить onPause()
9. Если B открыта с флагом FLAG_ACTIVITY_NO_HISTORY, то A всегда будет пересоздана(onCreate()) при Back из B

## 11. Выберите верные утверждения, если система убила процесс приложения и вы заходите в это приложение через недавно открытые?

## 21. Что будет, если в Hashmap положить элемент с ключом null?

1. HashMap разрешит один ключ null, значение сохранится, повторная вставка с null перезапишет значение
2. Будет NullPointerException
3. Null-ключи запрещены, компилятор не даст скомпилировать код
4. HashMap автоматически заменит null на пустую строку ""
5. Можно добавить несколько null-ключей, они будут храниться в разных бакетах
6. Null-ключ добавится, но get(null) всегда будет возвращать null

## 22. Какие утверждения о структурах данных верные?

1. В динамическим массиве вставка в конце имеет амортизированную сложность O(1)
2. В односвязном списке поиск элемента по значению имеет сложность O(1)
3. В хэш-таблице средняя сложность поиска O(1), но в худшем случае может быть O(n)
4. В сбалансированном бинарном дереве поиска гарантируется O(log n) для поиска, вставки и удаления
5. Удаления первого элемента в динамическом массиве всегда O(1)
6. В двусвязном спике удаление элемента по известному узлу может быть выполнена за O(1)

## 23. Какие утверждения верны для графа с E ребрами и V вершинами?

1. При представлении графа в виде матрицы смежности проверка наличия ребра между двумя вершинами выполняется за O(1)
2. При представлении графа в виде списка смежности проверка наличия конкретного ребра всегда выполняется за O(1)
3. Обход графа в глубину имеет временную сложность O(V + E)
4. Обход графа в ширину в худшем случае посещает каждую вершину и каждое ребро только один раз
5. В любом ориентированном графе без циклов всегда существует единственный возможный топологический порядок
6. Если граф представлен матрицей смежности, то по памяти он всегда эффективнее списка смежности

## 24. Что произойдет, если hashcode вернет одинаковое значение для разных объектов, но equals для этих элементов разный при добавлении нового элемента в hashMap?

1. Один объект перезапишется другим
2. HashMap сохранит оба объекта в одном bucket-e
3. HashMap сохранит объекты в разных bucket-ах
4. Нарушен контракт equals и hashcode - equals должен возвращать true, объект перезапишется другим

## 25. Что такое capacity в HashMap?

1. Количество элементов, которые уже добавлены
2. Размер массива bucket'
3. Максимальное количество ключей без коллизий
4. Порог заполнения, при котором происходит перераспределение ключей в HashMap

## 26. Когда бинарный поиск по массиву эффективен?

1. Когда массив отсортирован и содержит четное количество элементов
2. Когда поиск происходит по массиву, в котором все элементы отсортированы
3. Когда массив частично отсортирован. Например, каждая половина отсортирована отдельно
4. Когда массив отсортирован и содержит нечетное количество элементов

## 27. Какой контракт существует между hashCode и equals?

1. Если a.equals(b) == true, то a.hashCode() должен быть равен b.hashCode()
2. Если a.hashCode() == b.hashCode(), то a.equals(b) должен быть true
3. HashCode должен быть уникальным для каждого объекта
4. Equals() должен сравнивать ссылки, а не поля

## 34. Выберите верные утверждения о модификаторе inline?

1. Тело inline функции и переданных лямбд может быть встроено в место вызова, уменьшая накладные расходы на лямбды
2. Refied параметры возможны только у inline функций
3. Inline всегда уменьшает размер байт-кода и ускоряет выполнение
4. Crossinline запрещает нелокальные return из лямбды переданной в inline
5. Inline функции нельзя использовать с generic параметрами
6. Inline позволяет использовать non-local return из лямбды
7. Если лямбда-параметр помечен как noinline, то он не будет встроен вместо вызова
8. Ключевое слово crossinline запрещает non-local return из лямбды

## 35. Что означает ключевое слово <out T>?

1. T можно использовать только как входной параметр (consumer only)
2. T можно возвращать / производить (producer only) - это ковариантность
3. T гарантирует потокобезопасность
4. T отключает стирание типов на JVM
5. T можно использовать и для входных и для выходных параметров без ограничений
6. Out запрещает использовать T в параметрах функций внутри интерфейса
